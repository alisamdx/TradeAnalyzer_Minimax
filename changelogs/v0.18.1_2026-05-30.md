# v0.18.1 — 2026-05-30

## Bug fix: atm_iv stored as decimal fraction instead of percentage

### Root cause
The project convention (see `CLAUDE.md`) is that IV is stored and displayed as a **percentage** (28.5, not 0.2850). Two separate capture paths both violated this:

1. **IVolatility path** — `IVxRow.iv30` is a decimal fraction (`0.285 = 28.5%`) as returned by the API. `IvHistoryService.backfillIvHistory()` passed `row.iv30` directly to `storeReading()`, writing `0.285` instead of `28.5`.

2. **E*Trade auto-capture path** — `OptionContract.iv` is already a percentage (`28.5`). `captureFromEtradeChain()` was explicitly dividing it by 100 (`c.iv / 100`) before passing to `computeAtmIv()`, producing a decimal result (`0.285`) that was then stored as-is.

Both paths resulted in the `Curr IV` column in the Opportunity Dashboard showing `0.3%` instead of `30%`, and corrupted IV rank calculations (since min/max/current were all tiny fractions).

### Fixes

| File | Change |
|------|--------|
| `src/main/services/iv-history-service.ts` | IVolatility path: `storeReading(ticker, row.date, row.iv30 * 100, ...)` — multiply by 100 before storing |
| `src/main/services/iv-history-service.ts` | E*Trade path: `iv: c.iv` — removed erroneous `/ 100` so `computeAtmIv` operates in percentage space |
| `migrations/017_fix_iv_atm_pct.sql` | `UPDATE iv_history SET atm_iv = atm_iv * 100` — converts all existing rows |

### Verification
- `npm run typecheck` passes clean
- After migration runs, `Curr IV` in Opportunity Dashboard shows e.g. `30%` not `0.3%`
- IV rank/percentile calculations now operate on correct 0–100% scale
