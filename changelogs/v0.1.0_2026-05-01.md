- **Project scaffolding**: Electron 32 + React 18 + TypeScript 5 via `electron-vite`. Strict TS across main/preload/renderer. ESLint 9 + Prettier 3. Vitest 2 for tests.
- **Section 7 (Engineering Practices) infrastructure**:

  - `/migrations/` numbered SQL migration system; current head = `001_init.sql`.
  - `/logs/api/` and `/logs/errors/` runtime directories (gitignored, created on first launch). Wiring of the actual log writer is deferred to Phase 2 when the first outbound API call ships.

  - `.gitignore`, `.env.example`, `.prettierrc`, `eslint.config.js`.
  - `AI_CONTEXT.md` and `AI_PROMPT.md` at repo root.
- **FR-1 — Watchlist Management**:
  - SQLite schema with `watchlists`, `watchlist_items`, `settings`, `schema_version` tables.
  - Case-insensitive uniqueness on watchlist names (unique index on `lower(name)`).
  - `Default` watchlist auto-created on first launch; renamable, undeletable.
  - Service layer (`WatchlistService`) implements: create / rename / delete, list all, list items, add ticker (with duplicate-rejection per list), bulk add, remove ticker(s).
  - CSV export: columns `ticker, added_date, notes`.
  - CSV import: required `ticker` header, optional `notes` and `added_date`. Returns `{ imported, skipped: [{row, reason}] }` so the UI can render the spec-required summary dialog.
  - Round-trip export → import is identical (FR-1 acceptance criterion).
  - IPC surface `window.api.watchlists.*` exposed via context-isolated preload bridge.
  - Minimal React UI: sidebar of watchlists, main pane with items table, create/rename/delete actions, ticker add/remove, CSV import/export buttons.
- **Tests**: 30+ Vitest cases covering service CRUD, name-uniqueness rules, default-list invariants, CSV parser edge cases, and migration runner. Runs in <5s offline.

## AI session notes

- **Polygon-dependent parts of FR-1 are intentionally deferred.** FR-1.4 (autocomplete from S&P 500 universe), FR-1.7 (per-row last price / day % / volume / sector / auto-refresh) all need the DataProvider, which lands in Phase 2. The UI shows ticker + notes + added_date in the row for now; the price/volume/sector columns exist in the table header as `—` placeholders to make the future fill-in obvious. Don't delete the placeholder columns when adding the data; just bind them.
- **Why a unique index on `lower(name)` and not `COLLATE NOCASE`**: preserves the user's chosen casing on display ('Tech Megacaps', not 'tech megacaps') while still rejecting case-only duplicates. `COLLATE NOCASE` on the column would have made comparison operators case-insensitive elsewhere in queries.
- **DB driver pivot — `node:sqlite`, not better-sqlite3.** Initially specced as better-sqlite3, but the dev box has no Visual Studio Build Tools and better-sqlite3 12.x has no Node-24 prebuild yet. Switched to Node 24's built-in `node:sqlite` and wrapped it in `src/main/db/connection.ts` with `openDatabase` + `withTransaction`. The wrapper is the swap point — moving back to better-sqlite3 is changing this one file. API differences captured in `AI_CONTEXT.md → Known Quirks`.
- **Vite/Vitest can't statically resolve `node:sqlite`.** Static `import { DatabaseSync } from 'node:sqlite'` makes vite's resolver strip the `node:` and try to load a missing `sqlite` package. Workaround: `createRequire(import.meta.url)('node:sqlite')` in `connection.ts`. Don't undo this without verifying tests still load.
- **Windows file lock during install.** When iterating on dependencies, an `EBUSY` lock on `node_modules/better-sqlite3` (probably Windows Defender holding a transient handle on the empty folder after a failed gyp run) blocked `rm -rf node_modules`. Workaround used: install in a sibling directory (`TradeAnalyzer-work`) with `mklink /J` junctions, run tests there, wait for the lock to clear, then reinstall in the source. The lock cleared after ~5 minutes. If it happens again: don't fight it, install in a parallel dir.
- **Logs directory wiring is deferred.** The folder structure is correct per EP-3 / EP-4 but the actual logger doesn't run yet because no API calls happen in Phase 1. Phase 2 must add the structured JSONL writer with daily rotation and key-redaction before the first Polygon call.
- **No pre-commit hook yet (EP-6.6 / EP-7.5).** Adding `husky` was tempting but adds an install step that may surprise downstream contributors; deferring to Phase 2 with a single PR that introduces husky + lint-staged + the test gate together. For now, run `npm test` manually before commits.
- **Verified results (this session):** `npm test` → 37 passed, 0 failed, ~11s total. `npm run typecheck` → both tsconfig.node.json and tsconfig.web.json pass. App not yet launched in Electron (would require electron-rebuild for native modules — not relevant since we're on `node:sqlite`, but I haven't run `npm run dev` to confirm window opens). That's a known gap for Phase 1 and the first thing to verify in Phase 2.
